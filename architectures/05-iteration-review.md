## 迭代1 回顾（Sprint Retrospective）

### 做得好的

1. **TDD 流程有效**：每个 Phase 先写测试再开发，62 个测试一次通过率高，大幅减少回归 bug
2. **分步查询策略**：发现 Supabase JS SDK 的嵌套查询兼容问题后，果断切换为分步查询，避免了更大的返工
3. **Phase 划分清晰**：按功能域划分 6 个 Phase，每个 Phase 独立可测试，进度可控
4. **实时通知架构**：Socket.io + HTTP API 混合方案，在线实时推送 + 离线拉取兜底，实现成本低且可靠

### 做得不好的

1. **代理问题浪费时间**：Windows 环境下 Node.js 无法直连 Supabase（本地代理），排查花了较长时间。**改进**：开发环境文档应提前记录网络配置要求
2. **前端 onShow 生命周期问题**：uni-app Vue 3 的 `onShow` 需要从 `@dcloudio/uni-app` 导入，写法与 Vue 2 不同，导致首页不加载数据。**改进**：使用新框架前先做 spike（技术预研）
3. **前端联调偏晚**：后端 API 全部完成后才测前端，发现问题时已经积累了。**改进**：迭代2 每个 Phase 完成后立即前端联调
4. **性能未优化**：首次加载延迟较高，信息流查询涉及多次数据库请求。**改进**：迭代2 考虑接口合并、缓存策略

### 改进措施（带入迭代2）

| 改进项 | 具体措施 |
|--------|---------|
| 前端联调前移 | 每个功能完成后立即浏览器验证 |
| 性能关注 | 信息流增加缓存，减少 Supabase 请求次数 |
| 技术预研 | 图片上传先做 spike 验证 Supabase Storage |
| 加载体验 | 添加骨架屏/loading 状态 |
